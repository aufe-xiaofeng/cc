连接在Internet上的主机名必须是惟一的吗？

这是肯定的。因特网不允许有两个（或更多的）主机具有同样的主机名。但是必须注意，这里所说的“主机名”指的是主机的“全名”(full name)，它也就是“主机的域名”，而不是指一个主机的“本地名字”。

例如，很多单位的网站服务器主机的本地名字都愿意取为www。这主要是为了便于记忆，使人一看见这www，就知道这个计算机是用来存放该单位网页信息的，使得人们可以利用HTTP协议来访问这个网站。所以当我们看到下面这样的网址：http://www.google.com就应当很明确，在整个因特网范围是惟一的主机名就是www.google.com。但应注意，主机名有两种，即全名和本地名字(local name)。虽然主机的全名在因特网上必须是惟一的，但主机的本地名字只需要在本级域名下是惟一的即可。例如，“.google”是在顶级域名“.com”下注册的二级域名。www是这个主机在二级域名“.google”下的本地名字。全世界有很多的主机使用相同的本地名字（例如，www或mail），但这并不会产生混乱。我们可以看出，如果google将其网站主机的本地名字取为其他的名字xyz，那么它的网址就要变成：http://xyz.google.com

但这样做并没有什么好处，只能给别人增加一些记忆上的麻烦。我们还要指出，虽然主机名在因特网中必须是惟一的，IP地址在因特网中也必须是惟一的，但一个主机名却可以对应多个IP地址。

简述电子邮件的工作过程。

FTP采用客户/服务器工作模式，在客户与服务器之间利用TCP建立可靠连接，但与一般客户/服务器模式不同的是，FTP客户端与服务器之间要建立双重TCP连接，一个是控制连接，另一个是数据连接。控制连接传输控制信息（文件传输命令）；数据连接则是真正用来传输文件数据。FTP服务是一种交互式服务，客户每次调用FTP便与服务器建立一个会话，并利用控制连接向服务器发送控制命令来维持会话，直到FTP关闭。FTP客户进程每提出一个请求，服务器便与客户建立一个数据连接，进行文件数据传输，一旦数据传输结束，数据连接就被撤除。

为了满足多FTP客户请求的需要，FTP服务器采用并发服务器方式。FTP服务器进程由两大部分组成：一个主进程，负责接收新的请求；另外有若干个从属进程，负责处理单个请求。主进程的工作步骤如下：

（1）打开熟知端口（端口号为21），使客户进程能够连接上；

（2）等待客户进程的连接请求；

（3）启动从属进程处理客户进程的请求。从属进程对客户进程的请求处理完毕后即终止，但从属进程在运行期间根据需要还可能创建其他一些子进程；

（4）回到等待状态，继续接受其他客户进程的请求。主进程与从属进程的处理并发进行。

当客户进程向服务器进程发出建立连接请求时，要寻找服务器进程的熟知端口21。客户进程为自己选择一个TCP端口号，主动在这个端口与服务器21号端口之间建立控制连接。同时还要通过控制连接告诉服务器进程自己的另一个端口号码，用于建立数据连接。当控制连接建立以后，服务器主进程要为此请求建立一个从属进程，这个从属进程就是一个控制进程。当控制进程接收到FTP客户发送来的文件传输请求后，就创建一个“数据传输进程”和“一个数据连接”。并将数据连接连接到数据传输进程。当文件传输完毕后关闭“数据连接”。

什么叫域名解释？如何对域名进行解释的？

由于域名的意义明确 , 容易记忆 , 因此在应用层 , 用户都喜欢用域名。而 TCP 层和 IP 层都用 IP 地址，因此，存在从域名到 IP 地址的地址映射，这就叫做域名解释。域名解释的过程：当解释器要解释一个域名时，它就成为域名服务器的客户，并送出一个域名询问报文给本地域名服务器，本地域名服务器找到对应的 IP 地址后送回一个域名回答报文。

而本地域名服务器进行域名解释的方式有两种：

（ 1 ）递归解释：域名询问一次，本地域名服务器能解释就解释，不能解释的话，就交给根服务器逐个节点进行求解。

（ 2 ）反复解释：域名询问一次，本地域名服务器能解释就解释，不能解释的话，本地域名服务器就暂时成为另一个域名服务器的客户，去询问该服务器， …… 如此反复进行，直到得到结果为止。

什么说 IP 数据报传输既快捷，又能适应大规模网络的传输？

因为 IP 数据报的传输是 “ 尽力传递 ” 机制，即只管将数据报尽快传往信宿机，无论传输对与否，不做检验，不发确认，所以 IP 数据报传输快捷。 IP 数据报是采用无连接的数据报机制，是分组交换的一种扩展。它允许发送机将一个个数据分组传输通过 Internet 。每个分组都包含有信宿地址的信息，而每个分组的传输是完全独立的。信源主机产生一个分组，并把信宿地址放在分组头上，然后将分组发送给附件的一个路由器。但路由器收到分组后，它根据分组的信宿地址，选择下一个路由器，并把分组传送给它。这样，一个路由器接一个路由器，最后，把分组传输到与信宿机相连的路由器上，并把它传输给信宿机。通过这样的传输机制， IP 数据报就能适应大规模网络的传输。

数据报采用 “ 尽力传递 ” （ Best-effort delivery ）的含义是什么？

尽力传递是 IP 协议定义的一种通信语法，这种方式不需要任何的差错检测和流量控制。尽力传递方式可以把数据报尽快地传送到信宿中去，但它并不保证数据报那能对地到达信宿。它可能出现下面一些问题：

（ 1 ）数据报重复：由于每个数据报是独立寻径的，可能有些数据报超时了还没有到达信宿。信源以为丢失了，重发数据报。实际上该数据报可能并没有丢失，只不过迟到而已，因此，造成数据报重复。

（ 2 ）分组传输延迟或不按顺序：同样，由于每个数据报是独立寻径的，有些数据报可能先发送后到达，因而造成分组不是按顺序到达。

（ 3 ）数据损坏：数据在传输过程中，由于噪音的干扰，有些数据可能被损坏，因而造成整个数据报都不能用。

（ 4 ）数据报丢失：数据报在传输过程中，由于拥塞，有些数据报可能被路由器丢失。
